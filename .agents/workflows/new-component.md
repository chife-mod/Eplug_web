# New Component Workflow

При добавлении нового компонента, функционального блока или значимой фичи — следуй этим шагам строго по порядку.

---

## Куда писать документацию?

Это ключевое правило структуры проекта:

| Тип | Папка | Примеры |
|---|---|---|
| **Логика / Функциональный блок** | `docs/architecture/` | роутинг, data flow, скрипт, модуль JS, паттерн |
| **UI-компонент / Страница / Сборка** | `docs/components/` | кнопка, карточка, хедер, wireframe-страница, секция |

> **Правило:** Если это «как это работает» → `architecture/`. Если это «как это выглядит и используется» → `components/`.

---

## 1. Документация первая (обязательно)

Перед написанием кода создай файл документации.

### Для UI-компонента → `docs/components/{name}.md`

```markdown
# {Component Name}

**Файл:** `path/to/file.ext`  
**Тип:** UI-компонент / Страница / Сборка  
**Роль:** Одна строка — что это и зачем

## Описание
Что делает этот компонент, как выглядит, в каком контексте используется.

## Структура / Состав
Из каких частей состоит (вложенные элементы, секции).

## API / Параметры
Классы, data-атрибуты, JS-хуки или входные данные (если есть).

## Зависимости
Что использует этот компонент. Кто зависит от него.

## Известные ограничения
Edge cases, браузерные особенности, незавершённые части.
```

### Для логического/функционального блока → `docs/architecture/{name}.md`

```markdown
# {Block Name}

**Файл(ы):** `path/to/file.js`  
**Тип:** Функциональный модуль / Паттерн / Data flow  
**Роль:** Одна строка — что это делает

## Описание
Какую задачу решает этот блок в архитектуре.

## Принцип работы
Пошагово: как работает логика (алгоритм, поток данных).

## Зависимости
Что нужно для работы. Кто использует этот блок.

## Известные ограничения
Ограничения, edge cases, что ещё нужно сделать.
```

## 1.5 Регистрация на Project Hub (обязательно, сразу после документации)

После создания doc-файла — **сразу** добавить карточку в дашборд `index.html` (корень проекта).

### Правила карточки:
1. **Секция**: `<!-- Isolated Components -->` → в `.hub-grid`
2. **Класс ссылки**: `eplug-tile hub-animate dynamic-app-link`
3. **`data-app-path`**: React-роут компонента (например, `/test-principles-animated`)
4. **Описание** — строго одна строка (не переносить! ломает выравнивание прогресс-баров)
5. **Статус**: `active` (in-progress) или `done` (завершён)
6. **Прогресс**: `in-progress` fill — фиолетовый градиент; `done` — зелёный
7. **Путь в footer**: имя папки компонента, например `OurPrinciplesAnimatedSlider`
8. **Обновить счётчик**: `hub-section-count` → +1

> **Правило:** Компонент без карточки на дашборде = компонент не существует для команды. Регистрируй всегда.

---

## 2. Реализация

Пишем код. Если в процессе что-то изменилось — обновляем doc-файл.

---

## 3. Обновление архитектурного overview (если нужно)

Если компонент меняет общий поток данных или вводит новый паттерн:
- Обновить `docs/architecture/overview.md` (таблицу функциональных блоков или структуру)

---

## 4. История изменений

Добавить запись в `history/HISTORY.md`:
```markdown
## YYYY-MM-DD — {Краткое название}

### {Компонент / Активность}
- Что именно сделано.
- Какой документ создан/обновлён.
```

---

## 5. Коммит вместе

Документация и код — всегда в одном коммите:
```bash
git add docs/ history/ && git add {файлы компонента} && git commit -m "feat: {name} + docs"
```
